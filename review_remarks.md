# Отчет о проверке приложения "Протоколы ЭБ" (v0.1.0)

Этот отчет содержит результаты всестороннего анализа кодовой базы, пользовательского интерфейса (UI/UX) и функционального соответствия требованиям проекта (ТЗ). Анализ проведен с трех ролей: Senior Fullstack Developer, Senior UI Designer и Product Manager (PM).

---

## 1. Senior Fullstack Developer: Анализ кода, архитектуры и ошибок

### 🔴 Критические ошибки и баги (блокируют работу приложения)

1. **Блокировка базы данных при синхронизации (Windows File Lock)**
   - **Где:** [app/main.js](file:///d:/PythonProject/eb-protocols-v0.1.0/app/main.js#L212-L226) и [backend/sync/syncService.js](file:///d:/PythonProject/eb-protocols-v0.1.0/backend/sync/syncService.js#L92-L124)
   - **Суть:** В обработчике IPC `sync:download` открывается соединение с БД (`const db = openDb()`)... и удерживается открытым. Метод `syncService.downloadDb` пытается скачать файл с FTP и перезаписать его напрямую по пути `dbPath` (`eb_protocols.db`). На Windows открытое соединение SQLite монопольно блокирует файл БД. Попытка перезаписи файла завершится ошибкой `EBUSY` или `EPERM`, делая скачивание базы невозможным.
   - **Решение:** Закрывать активное подключение к БД перед началом скачивания и открывать заново только после успешной замены файла.

2. **Полное несовпадение прав UI с правами из БД/Excel**
   - **Где:** [app/renderer/index.html](file:///d:/PythonProject/eb-protocols-v0.1.0/app/renderer/index.html#L1377) и [backend/import/excelImporter.js](file:///d:/PythonProject/eb-protocols-v0.1.0/backend/import/excelImporter.js#L100-L109)
   - **Суть:** В UI список прав `allRights` жестко зашит на русском языке без знаков препинания (например, `'ответственный руководитель работ'`, `'оперативные переговоры'`). В Excel-файлах и БД права хранятся ровно так, как они написаны в заголовках колонок — сокращенно и с точкой с запятой на конце (например, `'отв рук-ль работ;'`, `'опер. переговоры;'`). При открытии карточки сотрудника UI делает проверку `sel.includes(r)`, которая всегда возвращает `false` из-за несовпадения строк. В результате **все чекбоксы прав в форме всегда остаются пустыми**, а при сохранении старые права затираются или сохраняются некорректно.
   - **Решение:** Внедрить маппинг (нормализацию) ключей прав между БД и UI.

3. **Пропуск обновления ревизии (Sync Bypass) при удалении записей**
   - **Где:** [app/main.js](file:///d:/PythonProject/eb-protocols-v0.1.0/app/main.js#L287-L312)
   - **Суть:** Обработчики удаления `employees:delete`, `references:deleteDepartment`, `references:deleteChairman` и др. напрямую выполняют SQL-запросы `UPDATE ... SET status = 'deleted'`, но **не вызывают** `touchDbModified(db)`. Из-за этого при удалении сотрудника или справочника `sync_revision` и `last_modified_at` в таблице `sync_state` не изменяются. Синхронизация посчитает, что база не менялась, и удаление никогда не улетит на FTP/другим клиентам.
   - **Решение:** Вызывать `touchDbModified(db)` во всех IPC-обработчиках удаления.

4. **Утечка файловых дескрипторов БД при ошибках импорта Excel**
   - **Где:** [backend/import/excelImporter.js](file:///d:/PythonProject/eb-protocols-v0.1.0/backend/import/excelImporter.js#L68-L153)
   - **Суть:** Соединение `db` открывается в начале функции `importExcel`, а `db.close()` вызывается только в самом конце. Если внутри транзакции или при чтении файла произойдет ошибка, выполнение прервется, и `db.close()` никогда не выполнится. Приложение будет накапливать открытые дескрипторы файлов БД.
   - **Решение:** Обернуть вызовы в блок `try ... finally { db.close(); }`.

5. **Регистрозависимый поиск Cyrillic (русских ФИО) в SQLite**
   - **Где:** [backend/db/repositories/employeesRepository.js](file:///d:/PythonProject/eb-protocols-v0.1.0/backend/db/repositories/employeesRepository.js#L80)
   - **Суть:** Запрос поиска использует `LOWER(last_name) LIKE LOWER(?)`. Но встроенная в SQLite функция `LOWER()` по умолчанию работает только с ASCII (латиницей). Для кириллицы регистр не изменится: `LOWER('Иванов')` вернет `Иванов`. Поиск становится регистрозависимым для русских букв.
   - **Решение:** Зарегистрировать в `better-sqlite3` кастомную функцию `LOWER` с поддержкой Unicode или использовать `COLLATE NOCASE` (хотя для кириллицы в стандартном SQLite это также требует настройки локали/расширений).

### 🟡 Архитектурные замечания и код-стайл

1. **Непоследовательный расчет даты следующей проверки (Off-by-one error)**
   - **Где:** На бэкенде (`protocolService.js`) дата следующей проверки рассчитывается как `ДатаПроверки + Период - 1 день` (например, `2026-06-22` + 1 год = `2027-06-21`).
   - В UI (`calcNextDate` в `index.html`) дата рассчитывается как `ДатаПроверки + Период` (например, `2027-06-22`). Это расхождение запутывает пользователей и ломает логику автоматического заполнения.
2. **Дублирование бизнес-логики**
   - Функции автонумерации протоколов (`getNextProtocolNumber`, `assertProtocolNumberAvailable`, `getProtocolYear`) продублированы и в [backend/protocol/protocolNumbering.js](file:///d:/PythonProject/eb-protocols-v0.1.0/backend/protocol/protocolNumbering.js), и в [backend/db/repositories/journalRepository.js](file:///d:/PythonProject/eb-protocols-v0.1.0/backend/db/repositories/journalRepository.js).
3. **Частый оверхед на I/O базы данных**
   - Каждая операция (даже мелкий поиск) открывает и закрывает файл БД заново через хелпер `withDb`. Для Electron-приложения эффективнее открыть одно подключение при старте и закрывать его при выходе.

## 2. Senior UI Designer: Анализ интерфейса и UX

### 🔴 Проблемы верстки и адаптивности

1. **Критический баг: Переполнение и недостижимость кнопок в модальном окне**
   - **Суть:** Окно "Добавить работника" содержит 11 полей ввода, его высота составляет около 650px. У элемента `.modal` нет свойства `max-height` и `overflow-y: auto`. На ноутбуках с небольшим экраном (например, 1366x768) или при изменении размера окна Electron по высоте, нижняя часть модального окна и кнопка "Сохранить" уходят под нижний край экрана, и прокрутить их невозможно. Пользователь не может добавить сотрудника.
   - **Решение:** Задать `.modal { max-height: 90vh; overflow-y: auto; }`.

2. **Сломанный адаптив бокового меню (1100px - 1200px)**
   - **Суть:** При ширине окна от 1100px до 1200px левый сайдбар схлопывается наверх (`grid-template-columns: 1fr`), но список навигации `.nav` все еще отображается вертикальным столбцом. В итоге верхняя часть приложения занята огромным пустым блоком меню. Только ниже 1100px меню перестраивается в сетку из 3 колонок.
   - **Решение:** Синхронизировать медиа-запросы сайдбара и навигации на единый брейкпоинт (1200px).

3. **Сжатие и наложение текста кнопок меню на малых экранах**
   - **Суть:** При ширине экрана менее 1000px меню навигации перестраивается в `repeat(3, 1fr)`. Кнопки имеют свойство `white-space: nowrap` (унаследованное от базового `button`). Названия кнопок (например, "📝 Новый протокол", "⚙️ Справочники") не помещаются по ширине, обрезаются или накладываются друг на друга.

4. **Бесконечно растягивающийся «бумажный» предпросмотр**
   - **Суть:** Блок предпросмотра протокола (`.preview`) имитирует лист бумаги А4 (белый фон, черные линии). Однако его ширина задана как `100%` от колонки. На мониторах Full HD колонка растягивается почти до 800px, делая строки текста слишком длинными и неестественными для документа.
   - **Решение:** Ограничить максимальную ширину предпросмотра (`max-width: 650px`) и выравнивать его по центру.

5. **Неконтролируемая ширина поисковой строки Suggest**
   - **Суть:** Выпадающий список автозаполнения (`.suggest`) имеет стили `width: max(100%, 520px);`. В узких окнах (или при сжатии панели) этот блок вылезает вправо за пределы контейнера, ломая сетку и перекрывая соседние элементы.

### 🟡 Визуальные недоработки и юзабилити (UX)

- **Отсутствие обратной связи кнопок в таблицах:** Кнопки действий (Удалить, Протокол) в таблицах используют класс `ghost` без четких hover/active состояний. При наведении цвет меняется незначительно, нет ощущения кликабельности.
- **Шрифты по умолчанию:** Приложение не использует современные веб-шрифты (Inter/Roboto), полагаясь на системные шрифты Windows (Segoe UI), что удешевляет внешний вид.

## 3. Product Manager (PM): Соответствие требованиям и готовность продукта

### 🔴 Невыполненные функциональные требования (Core Issues)

1. **Нерабочий раздел "График проверки знаний" (Calendar/Schedule)**
   - **Статус:** ❌ **Не реализовано**
   - **Детали:** В HTML есть заголовок и кнопки экспорта графика, но блок `#calendar` абсолютно пустой. В `app.js` нет ни одной строчки кода, которая бы рендерила календарную сетку, группировала сотрудников по месяцам следующей аттестации или обрабатывала переключение годов. Кнопки экспорта в Excel/PDF в этом разделе также не имеют обработчиков.

2. **Отсутствие генерации PDF**
   - **Статус:** ❌ **Не реализовано**
   - **Детали:** В ТЗ заявлена генерация PDF с использованием LibreOffice (soffice). На практике:
     - В коде бэкенда нет вызовов CLI-команд LibreOffice или библиотек для конвертации docx -> pdf.
     - Кнопка "PDF" в панели предпросмотра выдает заглушку: `showToast('Макет: будет сформирован PDF')`.
     - Таблица журнала имеет текстовую колонку "DOCX · PDF", но это статический текст.

3. **Отсутствие интерфейса для настройки FTP и шифрования БД**
   - **Статус:** ❌ **Не реализовано**
   - **Детали:** База данных поддерживает шифрование, а бэкапы/синхронизация завязаны на FTP-сервер. В коде бэкенда есть репозиторий настроек и IPC-методы для сохранения паролей в keytar. Однако в UI **нет ни одного экрана или формы для настройки этих параметров** (полей ввода хоста FTP, логина, паролей БД). Обычный пользователь не сможет настроить синхронизацию или включить шифрование без прямого редактирования БД SQLite сторонним софтом.

4. **Нефункциональный справочник Комиссий**
   - **Статус:** ⚠️ **Реализован частично / Тупик**
   - **Детали:** Таблица комиссий выводится в справочниках как read-only. Нет кнопок для добавления/удаления комиссий. В форме "Новый протокол" поле комиссии — это просто текстовый инпут, куда пользователь вбивает название руками. Выбрать готовую комиссию из справочника и автозаполнить председателя и членов комиссии невозможно.

5. **Невозможность управлять правами сотрудника из UI**
   - **Статус:** ⚠️ **Реализован частично**
   - **Детали:** Добавить или изменить права работника (например, "допускающий", "работы на высоте") в его карточке невозможно — в модальном окне добавления сотрудника этих полей нет. Из формы протокола эти чекбоксы тоже не сохраняются в профиль сотрудника (только пишутся текстом в лог протокола).

## Итог проверки (PM Summary)

Приложение **не готово к релизу** и находится в состоянии **раннего прототипа (MVP с критическими недоработками)**. Основные функции экспорта (PDF), планирования (График) и настройки (FTP, шифрование) полностью отсутствуют в графическом интерфейсе или не написаны в коде. Имеются серьезные баги блокировки файлов на Windows, которые сломают синхронизацию при первом же тестировании.

3. **PM/Dev:** Дописать генерацию PDF через LibreOffice CLI и создать полноценный экран настроек (settings view) для FTP и шифрования.
